DisplayXR framebuffer-size smoke test

Probes Chrome WebXR's behaviour against the DisplayXR runtime: whether oversize requests clamp (Part 1), and whether mid-session layer rebuilds pick up new per-view dims when the rendering mode changes (Part 2).

Test procedure

  1. Start displayxr-service.exe. Make sure DisplayXR is the active OpenXR runtime.
  2. Click Run smoke test and accept the XR permission. The page runs an initial clamp probe and logs per-view viewports for the mode that was active at session start.
  3. Leave the session running. Focus the DisplayXR service compositor window and press V (or 1/2) once to cycle to a different rendering mode. You should see RENDERING_MODE_CHANGED in the service logs.
  4. Come back to this page and click Probe viewports. This does a single requestAnimationFrame and reports the current per-view viewports WITHOUT rebuilding the layer — tells us whether Chrome re-queries mid-frame, or caches dims at layer-construction time.
  5. Click Rebuild baseLayer. This constructs a new XRWebGLLayer and calls session.updateRenderState({ baseLayer }), forcing Chrome to re-query xrEnumerateViewConfigurationViews. Log reports the new FB dims and viewports.
  6. Cycle the mode again and repeat steps 4-5 as many times as useful.
  7. Click End session when done. Copy the full log back.

Interpretation: if Rebuild baseLayer produces new FB dims that match the newly-active mode's atlas (e.g. 3840×1080 for a 2 × 1 LeiaSR 3D mode, or 3840×2160 for a 1-view 2D mode), then phase 2 just needs an env-var gate in oxr_system.c to bypass the legacy-compromise branch — no widening needed, and legacy apps stay unchanged.
